# Логика и Бизнес-правила (Logic & Business Rules)

Этот документ описывает детальную бизнес-логику работы приложения, отделенную от общих спецификаций.

## 1. Сметы и Расчеты (Estimates & Calculations)

### 1.1. Умный калькулятор (Smart Calculator)
Логика ввода объемов работ. Пользователь вводит любые 2 значения из 3-х, третье рассчитывается автоматически.
*   **Формула:** `Total (Всего) = Me (Мы) + Partner (Контрагент)`
*   **Поведение:**
    *   Если введены `Total` и `Me` -> `Partner = Total - Me`
    *   Если введены `Total` и `Partner` -> `Me = Total - Partner`
    *   Если введены `Me` и `Partner` -> `Total = Me + Partner`

### 1.2. Округление и Форматирование (Rounding & Formatting)
Правила отображения численных значений во всем приложении и отчетах.
*   **Удаление нулей:** Лишние нули после запятой всегда удаляются (`10.00` -> `10`, `10.50` -> `10.5`).
*   **Работы (Works):**
    *   Суммы и Цены округляются до **целых чисел** (Integer).
*   **Материалы (Materials):**
    *   Суммы и Цены округляются до **2 знаков** после запятой.
*   **PDF/Отчеты:** Правила округления применяются строго перед генерацией документов.

### 1.3. Наценка (Markup)
*   **Диапазон:** 0% - 100%.
*   **Логика:** Наценка применяется только к *цене продажи* для клиента. Закупочная цена (для нас) остается неизменной.
*   **Отображение:** В смете материалы могут быть показаны "С наценкой", "Без наценки" (базовая цена) или "Без цен" (только количество).

## 2. Инженерная карта (Engineering Map)

### 2.1. Силовые щиты (ShieldGroup)
*   **Стандартный ряд модулей:** 2, 4, 6, 8, 12, 24, 36, 48, 60, 72, 96, 120, 144.
*   **Расчет модулей:** Автоматический подсчет занимаемого места (`modules_count`) на основе добавленных устройств.
*   **Нормализация:** Авто-замена кириллицы/формата при вводе (2п -> 2P, 16а -> 16A).
*   **Заметки (Notes):** Поле для произвольных текстовых заметок. Видимость: "Пульсирующая кнопка" в карточке щита.

### 2.2. Слаботочка и Мультимедиа (Multimedia)
Логика подбора корпуса на основе количества линий (lines).
*   **≤ 4 линий:** -> "24 модуля"
*   **5 - 10 линий:** -> "36 модулей"
*   **> 10 линий:** -> "Индивидуальный расчет"
*   **Заметки:** Поддерживаются (аналогично силовым).

### 2.3. LED Зоны (LedZone) - Логика подбора щита
Автоматическая рекомендация размера щита на основе количества трансформаторов (drivers/transformers).
*   **1-2 шт:** -> "Переместить в слаботочный щит"
*   **3-4 шт:** -> "24 модуля"
*   **5-9 шт:** -> "36 модулей"
*   **10-12 шт:** -> "48 модулей"
*   **13-15 шт:** -> "60 модулей"
*   **> 15 шт:** -> "Индивидуальный расчет"

### 2.3. Слаботочка и Мультимедиа (Multimedia)
Логика подбора корпуса на основе количества линий (lines).
*   **≤ 4 линий:** -> "24 модуля"
*   **5 - 10 линий:** -> "36 модулей"
*   **> 10 линий:** -> "Индивидуальный расчет"

## 3. Отчеты и PDF (Reports & PDF)

### 3.1. Типы отчетов
1.  **Работы (Works):**
    *   *Наши (Our):* `Общий объем - Доля контрагента`. Расчет суммы.
    *   *Контрагент (Partner):* Только доля контрагента. Заголовок "ТВОИ".
    *   *Заказчик (Client):* Полный объем работ.
2.  **Материалы (Materials):**
    *   *Без цен:* Только список и количество.
    *   *С ценами:* Базовые цены закупки.
    *   *С наценкой (+%):* Цены, увеличенные на Markup.

### 3.2. Финансовый монитор (Finance Dashboard)
Экран для отслеживания неоплаченных этапов проектов и финансовых настроек.

#### 3.2.1. Основные возможности
*   **Список проектов:** Отображение всех проектов с неоплаченными этапами.
*   **Группировка:** Каждый проект содержит список своих неоплаченных этапов.
*   **Обновление данных:** Pull-to-refresh (жест "потянуть вниз"). Данные автоматически обновляются при возврате на экран.
*   **Глобальные настройки:** Два текстовых поля для заметок ("Смета контрагента" и "Финансовые заметки"), сохраняются отдельно для всей системы.

#### 3.2.2. Отображение сумм контрагента
*   **Формат:** Четкое разделение "наша часть" и "общая сумма".
*   **Пример:** "200$" (наша часть) + "из 350$" (общая сумма включая контрагента).
*   **Стиль:** Основные суммы проекта отображаются **чисто черным цветом**. Дополнительная информация "из XXX$" — серым (11px).
*   **Логика:** Отображается только если `employer_amount > 0`.
*   **Шрифт:** 11px с font-weight 500 для лучшей читаемости.

#### 3.2.3. Даты изменения этапов
*   **Источник:** Поле `updated_at` из модели Stage (автоматически обновляется при любом изменении).
*   **Форматы отображения:**
    *   "Сегодня" — изменен сегодня.
    *   "Вчера" — изменен вчера.
    *   "3 дн. назад" — изменен менее 7 дней назад.
    *   "07.02.2026" — полная дата, если 7+ дней назад.
*   **Назначение:** Помогает видеть активность по проектам (свежие изменения vs застойные этапы).

#### 3.2.4. UI компоненты
*   **Плашка этапов:** Стильная плашка с иконкой слоев (`layers_outlined`), градиентным фоном и зеленой обводкой.
*   **Кнопка подтверждения:** `OutlinedButton.icon` с ярко-зеленой обводкой и hover-эффектом.
*   **Компактность:** Padding карточек уменьшен (10→8px). Все блоки имеют стандартизированные внешние отступы **16px**.
*   **Адаптивные поля:** Текстовые поля настроек растут по содержимому (`minLines: 2`, `maxLines: null`). Внутренние отступы — **14px горизонтально / 10px вертикально**.

#### 3.2.5. Сметы: Заметки и Примечания
*   **Default Logic (Materials):** В пустой смете материалов автоматически подставляется текст "Не учтен вводной кабель.".
*   **Save Button Logic:** Кнопка «Сохранить» для примечаний материалов скрыта, если текст совпадает с дефолтным и не редактировался. Кнопка появляется только при ручном изменении текста.
*   **Визуальное единство:** Блок заметок в сметах визуально мимикрирует под развернутую панель "Итого" (фон `primaryColorLight` с opacity 0.5).

#### 3.2.5. API и Backend
*   **Endpoint:** `/api/projects/unpaid_projects/` — возвращает список проектов с неоплаченными этапами.
*   **Finance Settings:** `/api/finance/` (GET список, PATCH `/api/finance/1/` для обновления).
*   **Модель Stage:** Добавлены поля `created_at` (auto_now_add) и `updated_at` (auto_now).
*   **Миграция:** `0020_add_stage_timestamps` для добавления timestamp полей.

## 4. Статистика (Statistics Logic)

### 4.1. Динамика работ (Work Dynamics)
График показывает объем выполненных работ в денежном эквиваленте (USD) во времени.
*   **Источники данных:** `Stage` (Этапы).
*   **Дата:** Используется поле `created_at` этапа.
*   **Сумма:** Используется `total_amount_usd` (Общая сумма этапа).
*   **Группировка:**
    *   **По дням (Daily):** При просмотре за месяц. Формат оси: `dd.MM.yyyy` (например, `15.01.2026`).
    *   **По месяцам (Monthly):** При просмотре за год или все время. Формат оси: `MM.yyyy` (например, `02.2026`).
*   **Обработка данных (Backend):**
    *   API возвращает данные, сгруппированные по датам (`YYYY-MM-DD` или `YYYY-MM`).
    *   Фронтенд адаптирует формат отображения (санитизация пробелов, форматирование `DateFormat`).

### 4.2. Финансовая воронка
*   **Paid (Оплачено):** Сумма всех завершенных/оплаченных этапов.
*   **Pending (В ожидании):** Сумма этапов в работе.
*   **Визуализация:** Горизонтальный бар с разделением цветов (Зеленый/Оранжевый).

## 4. Автоматизация: Инженерная карта -> Смета (Материалы)
Цель: Автоматическое наполнение вкладки материалов на основе данных из "Щитов" (ShieldGroup).

### 4.1. Алгоритм агрегации (Aggregation Logic)
Система сканирует все устройства во всех ShieldGroup текущего проекта.
1.  **Группировка:** Устройства группируются по уникальному сочетанию 3-х признаков:
    *   `device_type` (Тип устройства)
    *   `poles` (Количество полюсов)
    *   `rating` (Номинал)
2.  **Поиск в каталоге:** Для каждой группы формируется ключ поиска `CatalogItem`:
    *   `shield_{device_type}_{poles}` (например, `shield_circuit_breaker_1P`).
    *   Ищется товар с matching key.

### 4.2. Формирование позиций в смете
При найденном совпадении создается `EstimateItem` (Материал):
*   **Наименование (Name):** Составное. Название из каталога + Номинал из щита.
    *   *Пример:* "Автомат 1P" (Catalog) + " 16А" (Shield) = **"Автомат 1P 16А"**.
*   **Количество (Quantity):** Сумма всех физических устройств данной группы.
*   **Цена и Ед.изм:** Копируются из найденного `CatalogItem`.

### 4.3. Авто-подбор корпуса (Auto-Enclosure Logic)
Автоматический выбор щита на основе количества модулей, типа щита и типа монтажа.

1.  **Разделение типов (Separation):**
    *   **Силовые щиты:** Обрабатываются отдельно.
    *   **Слаботочные/LED щиты:** Обрабатываются отдельно (используют отдельный ряд товаров).
    *   *Пример:* Если есть силовой щит 24 мод. и слаботочный 24 мод., в смете будет **две разные строки**.

2.  **Тип монтажа (Mounting):**
    *   Учитывается параметр щита: `internal` (Встраиваемый) или `external` (Навесной).

3.  **Формирование ключа (Catalog Key):**
    *   Для **Силовых**: `shield_enclosure_{SIZE}_{MOUNTING}`
    *   Для **Слаботочных/LED**: `shield_media_enclosure_{SIZE}_{MOUNTING}`
    *   *Пример:* `shield_enclosure_24_internal`, `shield_media_enclosure_36_external`.

4.  **Поиск стандарта:** Ищется ближайший больший стандарт размера:
    *   Ряд: 2, 4, 6, 8, 12, 24, 36, 48, 60, 72, 96, 120, 144.

5.  **Лимит:** Если `modules_count > 144` -> Создается позиция "ВНИМАНИЕ: Индивидуальный расчет щита" (цена 0).

6.  **Fallback (Если не найдено):**
    *   Создается позиция "ВНИМАНИЕ: Не найден корпус в каталоге! ({SIZE} мод, {MOUNTING})".

## 5. Автоматизация: Материалы -> Смета (Работы)
Цель: Генерация списка работ на основе закупленных материалов.

### 5.1. Логика связей (Related Works)
Авто-генерация работ происходит при добавлении/обновлении материалов.
1.  **Проверка связи:** В `CatalogItem` (Материал) проверяется поле `related_work_item`.
2.  **Создание работы:** Если поле заполнено, система создает соответствующую запись во вкладке "Работы".
3.  **Количество:** Копируется 1 к 1.
    *   *Пример:* 100 метров кабеля (Материал) -> 100 метров монтажа кабеля (Работа).
4.  **Обработка дублей (Replacement Logic):**
    *   Если работа уже есть в смете -> Старая запись удаляется, и создается новая с актуальным количеством.

### 5.2. Агрегация Материалов в Работы (New Aggregation Logic)
Механизм объединения нескольких разных материалов в одну работу.
1.  **Ключ агрегации (`aggregation_key`):**
    *   В `CatalogItem` (Материал) задается поле `aggregation_key` (например, `agg_cable_standard`).
    *   Все материалы с одинаковым ключом суммируются по количеству.
    *   *Пример:* Кабель 3х1.5 (100м) + Кабель 3х2.5 (50м) -> Сумма 150м.
2.  **Поиск работы:**
    *   Система ищет `CatalogItem` (Работа), у которого `mapping_key` совпадает с `aggregation_key`.
    *   Создается одна строчка работы с суммарным объемом.

### 5.3. Таблица префиксов (Prefix Reference)
Для автоматизации и связей используются специальные префиксы в полях `mapping_key`:

| Префикс | Назначение | Пример |
| :--- | :--- | :--- |
| **`shield_`** | Интеграция Инженерной карты (Щитов, Устройств) в смету материалов. | `shield_circuit_breaker_1P`, `shield_enclosure_24_internal` |
| **`agg_`** | Агрегация (группировка) разных материалов в одну общую работу. | `agg_cable_standard`, `agg_cable_internet` |

## 7. Система Шаблонов (Template System)
Система использует 4 независимых типа шаблонов для максимальной гибкости и удобства.

### 7.1. Типы шаблонов
1.  **WorkTemplate (Шаблон работ):**
    *   Содержит набор работ (`CatalogItem` с типом `work`).
    *   Применяется к Этапу (Stage).
    *   **Логика:** Работы добавляются в смету. Если работа уже есть, количество суммируется.
    
2.  **MaterialTemplate (Шаблон материалов):**
    *   Содержит набор материалов (`CatalogItem` с типом `material`).
    *   Применяется к Этапу (Stage).
    *   **Логика:** Материалы добавляются. **Важно:** Применение шаблона НЕ вызывает автоматический пересчет работ (это делается отдельным действием).
    
3.  **PowerShieldTemplate (Шаблон силового щита):**
    *   Содержит конфигурацию устройств (`device_type`, `poles`, `rating`, `quantity`).
    *   Применяется к Силовому щиту (`Shield` type='power').
    *   **Логика:** Создаются физические устройства (`ShieldGroup`). Количество элементов в шаблоне превращается в соответствующее количество строк в щите.
    
4.  **MultimediaTemplate (Шаблон мультимедиа):**
    *   Для наполнения слаботочных щитов.
    *   Содержит список устройств (`name`, `quantity`).
    *   Применяется к Слаботочному щиту (`Shield` type='multimedia').
    *   **Логика:** Создаются записи в наполнении щита.

### 7.2. Логика замены (Clear & Apply Logic)
При применении любого шаблона действует принцип "Полная замена":
*   **Очистка:** Перед добавлением элементов из шаблона, текущий раздел (все работы этапа, все материалы или все наполнение конкретного щита) полностью очищается.
*   **Безопасность:** Каждое применение шаблона (или импорт из инженерки) требует **обязательного подтверждения** пользователем (`ConfirmationDialog`).
*   **Результат:** После применения шаблона в разделе остаются только те позиции, которые были в шаблоне.

### 7.3. Управление Шаблонами в UI (Lifecycle in UI)
1.  **Создание (Create):**
    *   Создание пустых шаблонов через админку (существующая логика).
    *   **"Сохранить как шаблон" (Save as Template):**
        *   Возможность сохранить текущий этап (Работы или Материалы) как новый шаблон.
        *   Возможность сохранить текущую конфигурацию щита (Силовой или LED) как новый шаблон.
2.  **Удаление (Delete):**
    - Прямое удаление шаблона из диалога выбора шаблонов.
    - Требует подтверждения пользователя.

## 8. Управление файлами (File Management)

### 8.1. Сохранение имен (File Naming)
*   **Original Name:** Система сохраняет оригинальное имя файла при загрузке в поле `original_name`. 
*   **Логика захвата:** Имя извлекается напрямую из объекта загруженного файла на этапе создания записи (`POST`), чтобы избежать искажений, вносимых файловым хранилищем (storage backend).
*   **Отображение:** UI приоритетно отображает `original_name`. Если оно отсутствует, используется имя из URL.

### 8.2. Физическое удаление (Cleanup Policy)
*   **Data Hygiene:** При удалении записи `ProjectFile` через API, файл физически удаляется из файловой системы сервера. 
*   **Реализация:** Переопределение `perform_destroy` в ViewSet обеспечивает вызов `instance.file.delete()` перед удалением объекта из БД.

### 8.3. Оптимизация и UX (Memory & UX)
*   **Thumbnail Decoding:** `cacheWidth: 300` для предотвращения OOM.
*   **Auto-Expand:** Категории файлов, содержащие **5 и менее элементов**, раскрываются автоматически при загрузке. Более крупные категории свернуты.
