# Шаблон промптов для AI-агента: веб-версия

## 1. Как использовать

Этот файл — заготовка для рабочих промптов AI-агенту по этапам веб-версии.

Идея простая:

- в начало каждого промпта вставляется общий блок правил;
- ниже добавляется конкретная задача текущей итерации;
- в конце можно добавить дополнительные ограничения или пожелания.

## 2. Общий блок для начала промпта

Ниже блок, который можно вставлять почти без изменений в начало каждого рабочего промпта.

```text
Работаем над веб-версией текущего проекта на существующей кодовой базе Flutter/Django.

Общие правила для этой задачи:

1. Не ломай существующую бизнес-логику приложения.
2. Не ухудшай уже работающие сценарии для Android и Windows без крайней необходимости.
3. По возможности переиспользуй текущую архитектуру и существующий код, а не создавай параллельную реализацию с нуля.
4. Все текстовые файлы и изменения должны сохраняться в UTF-8.
5. Если меняешь поведение, сначала проверь текущую реализацию в коде, а не делай предположения.
6. Для web-версии учитывай два сценария:
   - браузер на телефоне;
   - браузер на ПК.
7. Если появляются platform-specific отличия, выноси их аккуратно и минимально, не дублируя весь frontend без необходимости.
8. Не делай лишних рефакторингов вне задачи.
9. Если видишь смежную проблему, которая прямо мешает текущей задаче, исправь ее; если не мешает — просто кратко отметь в итогах.
10. После изменений обязательно:
   - проверить сборку или тесты, которые относятся к измененному участку;
   - кратко описать, что именно изменено;
   - отдельно указать риски, ограничения или что еще осталось недоделанным.

Формат работы:

- Сначала коротко проанализируй, какие файлы и участки кода затронуты.
- Затем внеси изменения.
- После этого запусти релевантные проверки.
- В финале дай краткий отчет:
  - что сделал;
  - какие файлы изменил;
  - что проверил;
  - какие есть риски / что осталось на следующий шаг.
```

## 4. Короткая версия для маленьких задач

Если задача небольшая, можно использовать сокращенную версию.

```text
Работаем над веб-версией текущего проекта на существующей кодовой базе Flutter/Django.

Важно:
- не ломай бизнес-логику;
- не ломай Android и Windows;
- используй текущую архитектуру;
- сохраняй изменения в UTF-8;
- учитывай браузер на телефоне и на ПК;
- не делай лишних рефакторингов;
- после изменений обязательно проверь сборку/тесты по затронутому участку;
- в финале кратко напиши, что изменил, что проверил и какие есть риски.

Задача:
[сюда вставляется конкретная задача]
```

## 5. Рекомендуемые готовые формулировки для конкретных этапов

### 5.1 Этап: web-routing

```text
Текущая задача:
Подготовь нормальную браузерную навигацию для веб-версии. Нужно проанализировать текущую навигацию приложения и внедрить web-friendly routing так, чтобы корректно работали переходы, обновление страницы, browser back и прямое открытие нужных экранов.
```

### 5.2 Этап: web API URL и backend web-конфигурация

```text
Текущая задача:
Подготовь корректную работу web-версии с backend. Нужно убрать хрупкие адреса для браузера, продумать web-friendly API URL и при необходимости внести изменения в конфигурацию backend/frontend для корректной работы в браузере на ПК и телефоне.
```

### 5.3 Этап: upload файлов для web

```text
Текущая задача:
Подготовь корректную загрузку файлов в проект для web-версии. Нужно проанализировать текущую реализацию upload, убрать desktop/mobile-зависимости там, где они мешают браузеру, и сделать web-safe сценарий загрузки файлов без поломки Android и Windows.
```

### 5.4 Этап: download/open/share файлов для web

```text
Текущая задача:
Подготовь корректные браузерные сценарии для файлов: открытие, скачивание, save as и share. Нужно сохранить существующую логику продукта, но адаптировать ее под ограничения браузера и не поломать уже работающие платформы.
```

### 5.5 Этап: PDF для web

```text
Текущая задача:
Подготовь web-friendly работу с PDF-сметами. Нужно проанализировать текущий экспорт и действия с PDF, затем адаптировать их для браузера: preview/open, download и share там, где это поддерживается.
```

### 5.6 Этап: desktop web UI

```text
Текущая задача:
Адаптируй основные экраны под браузер на ПК. Нужно пройтись по ключевым рабочим экранам и сделать так, чтобы интерфейс выглядел как полноценная рабочая веб-система, а не как растянутый мобильный экран.
```

### 5.7 Этап: mobile browser UI

```text
Текущая задача:
Адаптируй основные экраны под браузер на телефоне. Нужно убедиться, что интерфейс остается читаемым, удобным и кликабельным на узких экранах без поломки desktop web, Android и Windows.
```

### 5.8 Этап: финальный web-audit

```text
Текущая задача:
Проведи финальный технический аудит веб-версии по текущей кодовой базе. Нужно проверить сборку, ключевые пользовательские сценарии, оставшиеся browser-specific проблемы и составить список конкретных доработок или исправлений.
```

## 6. Что полезно добавлять в отдельные промпты вручную

Ниже необязательные, но полезные уточнения:

- "Сначала покажи короткий план, потом вноси изменения."
- "Не меняй визуальный стиль вне нужного участка."
- "Если найдешь несколько вариантов реализации, выбери минимально рискованный."
- "Если задача упирается в platform-specific ограничения браузера, явно объясни это в финале."
- "Если нужно обновить docs, измени соответствующий файл в `/docs`."

## 7. Практическая рекомендация

Лучше не делать один огромный промпт на весь этап веб-версии.

Лучше делать так:

1. один промпт = один крупный блок работ;
2. после ответа агента — ручная проверка;
3. потом следующий промпт.

Это даст намного более предсказуемый результат и снизит риск того, что агент начнет менять слишком много вещей сразу.
